# Takeshi's Data Castle: System Design & Architecture

## 1. Tools & Tech Stack
* **Frontend:** React (Vite), TypeScript, Tailwind CSS
* **Backend:** Node.js, Express.js
* **Database:** PostgreSQL (Hosted via Neon)
* **AI & External APIs:** Google Gemini 3 Flash SDK, NewsAPI

---

## 2. System Workflow
The application operates as a full-stack interactive game show utilizing a strictly typed, three-phase loop:
1.  **Context Generation:** The Express backend fetches real-time global headlines via NewsAPI and synthesizes them into a surreal game show obstacle using LLM generation.
2.  **Resource Allocation:** The player is presented with five randomized items queried from the PostgreSQL database and must draft a survival strategy using their selection.
3.  **AI Evaluation:** The strategy, item metadata, and obstacle context are processed by a 3-judge AI panel. The backend parses the LLM output into structured JSON to determine the round outcome.

---

## 3. Artificial Intelligence Implementation
The system leverages **Google Gemini 3 Flash** for two distinct architectural roles:
* **Dynamic World-Building:** Synthesizing unstructured external data (news) into creative, context-aware challenges via open-ended generation.
* **Deterministic Evaluation:** Acting as a structured rules engine. Through precise prompt engineering, the LLM is constrained to act as three distinct personas (**Master Takeshi**, **Chaos-Chan**, and **The Logic Bot**) and output a rigidly formatted JSON array. This bridges the gap between generative reasoning and the fixed data contracts required by the frontend.

---

## 4. Data Usage & Strategy
To fulfill the **"Data at Scale"** requirement, the system anchors AI creativity in a controlled data corpus:
* **The Corpus:** A PostgreSQL database seeded with over 100 unique, pre-generated items.
* **Gameplay Impact:** Item randomization (`ORDER BY random() LIMIT 5`) is offloaded to the database layer rather than generated by the LLM. This grounds the AI's reasoning in a structured schema, forcing the model to contextualize fixed physical properties against a dynamic obstacle to calculate scores.

---

## 5. Key Architectural Decisions
* **RESTful API Orchestration:** The backend acts as a secure middleware layer, aggregating multiple upstream services (Gemini, News, DB) to keep API keys hidden and the React client lightweight.
* **Prompt Chaining & Regex Parsing:** To handle the inherent unpredictability of LLMs, the system uses a custom Regex pipeline (`extractJsonArray`) to safely extract and validate JSON payloads from the AI's response.
* **Resilience & Graceful Degradation:** A custom exponential backoff controller handles API rate limits (HTTP 429/503). If services are unavailable, the system seamlessly degrades to local mock data (`mockJudgePanel`), ensuring zero downtime during the demo.
* **Phase-Based UI State Machine:** The frontend utilizes a strict state machine (`AppPhase`) to isolate rendering logic, manage asynchronous loading states, and prevent race conditions.
